SUPABASE MIGRATION + USER ID REMAP GUIDE
========================================

Purpose
-------
Use this guide when moving Sarvam Dental Clinic from one Supabase project to another.

Important Truth
---------------
When users are created in a new Supabase project, `auth.users.id` values change.
That is normal.

This affects app ownership columns such as:
- public.admin_profiles.user_id
- public.appointments.approved_by
- public.appointment_status_history.changed_by
- public.blocked_dates.created_by
- public.blocked_slots.created_by
- public.clinic_settings.updated_by

Customer appointment records remain usable.
The remap is mainly about preserving admin ownership history.

Recommended Migration Approach
------------------------------
Use a mapped migration:
1. create new Supabase project
2. apply schema
3. create new auth users
4. import app data
5. map old UUIDs to new UUIDs
6. update env vars and redeploy
7. re-enroll MFA

What To Export / Recreate
-------------------------
Schema and config:
- `supabase/schema.sql`
- Auth URL Configuration
- email provider settings
- TOTP MFA enabled state

Data:
- appointments
- appointment_status_history
- blocked_dates
- blocked_slots
- clinic_settings
- visitor_events if analytics history matters
- admin_profiles

Auth and roles:
- dentist and staff users from Supabase Auth
- matching `admin_profiles` rows

Step 1 - Apply Schema
---------------------
Run `supabase/schema.sql` in the new project.

Step 2 - Create Real Users In New Project
-----------------------------------------
Create dentist and staff users in Supabase Auth.
After creating them, collect their new UUIDs from Authentication -> Users.

Step 3 - Build UUID Mapping
---------------------------
Create a private map like this:

Old User Email          Old UUID                New User Email          New UUID
-------------------     -------------------     -------------------     -------------------
old-dentist@test.com    OLD_DENTIST_UUID        dentist@clinic.com      NEW_DENTIST_UUID
old-staff@test.com      OLD_STAFF_UUID          staff@clinic.com        NEW_STAFF_UUID

Step 4 - Import Data
--------------------
Import non-auth data into the new project.
Recommended order:
1. clinic_settings
2. appointments
3. appointment_status_history
4. blocked_dates
5. blocked_slots
6. visitor_events
7. admin_profiles

Step 5 - Remap User IDs
-----------------------
Update every admin-linked column using the old->new UUID mapping.

Example SQL pattern:

update public.admin_profiles
set user_id = 'NEW_UUID'
where user_id = 'OLD_UUID';

update public.appointments
set approved_by = 'NEW_UUID'
where approved_by = 'OLD_UUID';

update public.appointment_status_history
set changed_by = 'NEW_UUID'
where changed_by = 'OLD_UUID';

update public.blocked_dates
set created_by = 'NEW_UUID'
where created_by = 'OLD_UUID';

update public.blocked_slots
set created_by = 'NEW_UUID'
where created_by = 'OLD_UUID';

update public.clinic_settings
set updated_by = 'NEW_UUID'
where updated_by = 'OLD_UUID';

Repeat for each migrated admin user.

Step 6 - Verify Old UUIDs Are Gone
----------------------------------
Run checks such as:

select * from public.admin_profiles where user_id = 'OLD_UUID';
select count(*) from public.appointments where approved_by = 'OLD_UUID';
select count(*) from public.appointment_status_history where changed_by = 'OLD_UUID';
select count(*) from public.blocked_dates where created_by = 'OLD_UUID';
select count(*) from public.blocked_slots where created_by = 'OLD_UUID';
select count(*) from public.clinic_settings where updated_by = 'OLD_UUID';

Expected result:
- no remaining old UUID references in these tables

Step 7 - Recreate Admin Roles
-----------------------------
Ensure the new auth users exist in `public.admin_profiles` with correct roles:
- dentist -> `dentist`
- staff -> `staff`

Step 8 - Re-enroll MFA
----------------------
TOTP authenticators should be set up again in the new Supabase project.
Treat MFA re-enrollment as a normal migration step.

Step 9 - Update Hosting Environment Variables
---------------------------------------------
Set the new project values in hosting:
- SUPABASE_URL
- SUPABASE_ANON_KEY
- SUPABASE_SERVER_KEY

If legacy fallback is still used:
- SUPABASE_SERVICE_ROLE_KEY

Then trigger a fresh deploy.

Step 10 - Update Supabase Auth URL Configuration
------------------------------------------------
Recommended production Site URL:
- https://sarvamdentalclinic.netlify.app

Recommended Redirect URLs:
- https://sarvamdentalclinic.netlify.app
- https://sarvamdentalclinic.netlify.app/doctors-login.html
- http://localhost:8888/doctors-login.html
- http://localhost:8889/doctors-login.html

Step 11 - Smoke Test After Migration
------------------------------------
- public booking
- weekly off overlay
- blocked date/session rules
- dentist login
- staff login
- TOTP setup and verify
- forgot password
- reset password
- approve / reject
- Day Snapshot
- Book for Patient
- Control Panel
- Manage Users
- Current Access Users
- analytics page
- Back to Dashboard visibility for logged-in admins only

Old Test Users
--------------
If old test users are no longer needed:
- ban them in Supabase Auth
- move or clear known DB references
- if deletion still fails, banning is operationally acceptable

Risk Notes
----------
- Never place the anon key in the server-key env var.
- Never expose `SUPABASE_SERVER_KEY` in frontend code.
- TOTP-only login is not the normal Supabase MFA model; keep email/password + TOTP.
